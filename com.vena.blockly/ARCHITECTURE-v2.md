### 注脚二：三层稳定度分层

Blockly 模块在物理上沿一条单向依赖链分为三层，**稳定度从内到外递减**：

| 层 | 名称 | 稳定度 | 演化节奏 | 职责 |
|----|------|--------|---------|------|
| 内层 | **RuntimeEngine** | 最稳定 | 季度级 | 控制图 / 逻辑图的运行时驱动、节点执行、句柄机制、作用域链 |
| 中层 | **EditorIR** | 中等 | 月级 | 编辑期中间表示，承担 Blockly JSON ↔ 运行时数据的双向转换 |
| 外层 | **BlocklyFrontend** | 最易变 | 周级 | Blockly Web 编辑器、积木定义、工具箱、UI 交互 |

**依赖方向铁律**：

```
BlocklyFrontend ──→ EditorIR ──→ RuntimeEngine
```

**单向，不可反向。** RuntimeEngine 不知道 EditorIR 的存在；EditorIR 不知道 BlocklyFrontend 的存在。任何反向依赖（例如 RuntimeEngine 引用 EditorIR 类型，或 EditorIR 引用 Blockly 前端常量）都属于架构破坏，必须在评审时拦下。

这个分层的目的：把"用户每周都在改"的前端波动隔离在最外层，避免它震荡到"季度才动一次"的运行时核心。

# Infinity 逻辑编排架构白皮书

**文档版本**：v2.0（最终总结版）
**状态**：已定稿，可作为技术实现依据

## 一、核心设计哲学

**逻辑图是大脑（决策），控制图是身体（执行）。**

| 概念 | 一句话定义 | 本质 |
|------|-----------|------|
| **逻辑图** | **"怎么算"**——无状态的复合函数，单帧执行，输入→输出 | 数据流图 |
| **控制图** | **"怎么做"**——有状态的行为流程，跨帧执行，可中断可等待 | 执行流图 |
| **执行引擎** | 驱动控制图 Tick、调度逻辑图执行的运行时 | 图虚拟机 |

## 二、逻辑图（Logic Graph）

### 2.1 定义
逻辑图是一张由 **函数节点** 组成的数据流图，描述"如何计算一个值"或"如何做出一个判断"。
- 执行模式：**同步、单帧**
- 状态：**无状态**（每次执行从零开始）
- 输入：参数列表
- 输出：返回值（单个值或结构体）

### 2.2 节点类型
| 类型 | 职责 | 示例 |
|------|------|------|
| **函数节点** | 调用已注册的 UGC 函数 | `Add(A, B)`、`GetAttribute(unit, "hp")` |
| **常量节点** | 提供常量值 | `100`、`"fire"`、`true` |
| **变量节点** | 读取输入参数或上下文变量 | `$caster`、`$target` |
| **分支节点** | 条件选择执行路径 | `If(condition, true_branch, false_branch)` |
| **输出节点** | 标记逻辑图的返回值 | `return result` |

### 2.3 执行模型
- **数据流驱动**：节点有输入/输出引脚，连线表示数据的流向，而非执行顺序。
- 当一个节点的所有输入引脚都收到数据时，该节点自动执行。
- 执行结果通过输出引脚传递到下游节点。
> **逻辑图中没有串行/并行/循环等控制节点，只有数据依赖关系。执行顺序由数据依赖自动推导。**

### 2.4 生命周期
```
创建 → 绑定输入参数 → 求值（数据流执行） → 返回结果 → 销毁
```
逻辑图不持有任何状态，不跨帧执行，每次调用都是一次完整的"计算过程"。

### 2.5 与 Unity 的对应
```
逻辑图 = C# 的 Func<T1, T2, TResult> 委托链
逻辑图节点 = 委托链中的一个函数调用
逻辑图执行 = 调用委托，返回结果
```

## 三、控制图（Control Graph）

### 3.1 定义
控制图是一张由 **行为节点** 组成的执行流图，描述"如何完成一个持续性任务"。
- 执行模式：**异步、跨帧**
- 状态：**有状态**（维护执行位置、子任务状态）
- 输入：启动参数
- 输出：执行结果（完成/失败/中止）

### 3.2 节点类型
| 类型 | 职责 | 行为 |
|------|------|------|
| **动作节点** | 执行具体的游戏行为 | 执行持续动作，返回"完成/失败/运行中" |
| **控制节点** | 管理子节点的执行方式 | 串行/并行/分支/循环/等待/中止 |
| **条件节点** | 调用逻辑图并获取布尔结果 | 调用逻辑图，根据返回值决定走向 |
| **调用节点** | 启动另一个控制图 | 创建并启动子控制图，获取句柄 |
| **等待节点** | 挂起执行，等待条件满足 | 等待时间、等待事件、等待逻辑图返回 true |

### 3.3 执行模型
- **执行流驱动**：节点有执行输入/输出，连线表示执行顺序。
- 控制节点负责调度子节点的执行顺序和方式。
- 每个节点在 `Tick()` 中被驱动，返回 `Running/Success/Failure`。
- 控制图是"长运行"的，需要被持续驱动。

### 3.4 生命周期
```
创建 → 启动（执行第一个节点）→ Tick（每帧驱动）→ 完成/失败/中止 → 销毁
```
- **挂起**：遇到等待节点，挂起当前执行栈。
- **恢复**：等待条件满足后恢复执行。
- **中止**：外部请求中止，立即停止所有子节点并清理。

### 3.5 与 Unity 的对应
```
控制图 = MonoBehaviour 组件
控制图节点 = 组件内部的一段行为逻辑
控制图执行 = MonoBehaviour.Update() 驱动
控制图挂起 = 协程 yield return 或 状态机等待
```

## 四、两图的交互契约

### 4.1 交互方式总览
| 方向 | 方式 | 说明 |
|------|------|------|
| **控制图 → 逻辑图** | 同步调用 | 控制图节点调用逻辑图，立即获取返回值，继续执行 |
| **控制图 → 控制图** | 异步启动 | 控制图节点启动子控制图，立即返回句柄，不等待完成 |
| **逻辑图 → 控制图** | 异步操作 | 逻辑图节点通过"创建/访问/销毁"接口操作控制图，不持有 |

### 4.2 核心约束（不可违背）
> **逻辑图不能持有控制图，只能通过句柄间接操作。逻辑图不能修改控制图的内部结构。**

### 4.3 逻辑图对控制图的三种操作
| 操作 | 函数签名 | 说明 |
|------|---------|------|
| **创建** | `CreateBehavior(behaviorId, params) → handle` | 启动一个控制图实例 |
| **访问** | `GetBehaviorState(handle) → state` | 查询控制图状态（是否完成、进度、返回值） |
| **销毁** | `DestroyBehavior(handle)` | 强制终止并清理控制图 |

### 4.4 句柄机制
逻辑图通过"句柄"引用控制图，而非直接持有。句柄是一个值类型（如 GUID 或整数 ID），逻辑图本身依然是无状态的。
```
逻辑图节点示例：
[创建控制图: 播放施法动画] → handle
[查询控制图状态: handle] → isComplete
[条件分支: isComplete?]
  ├─ true → [执行后续逻辑]
  └─ false → [返回：等待中]
```

## 五、与 Blockly 的映射关系

你已经选择 **Blockly** 作为编辑器前端，这是完全正确的架构决策。

### 5.1 积木分类与图类型对应
| 积木颜色 | 对应图类型 | 功能 | 连接方式 |
|---------|-----------|------|---------|
| 蓝色（值积木） | 逻辑图 | 数值计算、属性读取、条件判断 | 值（Value）连接 |
| 绿色（流程积木） | 控制图 | 串行/并行/分支/循环/等待 | 语句（Statement）连接 |
| 橙色（行为积木） | 控制图 | 造成伤害、施加Buff、移动单位 | 语句（Statement）连接 |

### 5.2 约束在 Blockly 中的体现
| 约束 | 编辑器 UI 实现方式 |
|------|-------------------|
| 逻辑图工具箱里只有蓝色积木 | 不显示绿色和橙色积木 |
| 控制图工具箱里包含所有积木 | 显示绿色、橙色，以及可调用的蓝色积木 |
| 逻辑图不能持有控制图 | 不提供"持有控制图"的积木类型 |
| 逻辑图只能通过句柄操作控制图 | 提供"创建/查询/销毁控制图"的蓝色函数积木 |

### 5.3 完整技术方案
| 模块 | 技术实现 |
|------|---------|
| **前端编辑器** | Blockly Web 应用 |
| **数据导出** | Blockly 将积木图序列化为 JSON |
| **数据导入** | Unity 加载 JSON，构建逻辑图/控制图数据 |
| **运行时执行** | Unity 中的逻辑图引擎 + 控制图引擎驱动 |

## 六、与 ECA 架构的对比

| 维度 | ECA 触发器架构 | 逻辑图 + 控制图架构 |
|------|---------------|-------------------|
| **逻辑组织** | 扁平列表，按事件分散 | 结构化图，逻辑集中 |
| **分支逻辑** | 需要底层预置动作模板 | 用户可在图中自由组合 |
| **自定义性** | 受限于预置动作库 | 无限，用户可自由组合基础节点 |
| **复用性** | 差，逻辑绑定特定事件 | 好，逻辑图可被任意控制图复用 |
| **可维护性** | 随项目增大趋于混乱 | 随项目增大保持清晰 |
| **表达力** | 有限状态机 | 图灵完备 |

```
│                            ↓ 可以调用                          │
├─────────────────────────────────────────────────────────────────┤
│                        逻辑图（无状态、单帧）                    │

│  │ 数值计算（加减乘除、随机数）                               │   │
│  │ 条件判断（比较、逻辑运算）                                 │   │

│  │ 查询控制图状态（只读）                                     │   │

│  │ 不能等待控制图完成（等待是控制图的行为）                   │   │

```

## 八、核心设计原则总结

> **逻辑图是纯函数（Pure Function），控制图是有状态对象（Stateful Object）。**

| 维度 | 逻辑图 | 控制图 |
|------|--------|--------|
| 核心职责 | 决策与计算 | 执行与调度 |
| 执行模型 | 数据流驱动 | 执行流驱动 |
| 生命周期 | 单帧 | 跨帧 |
| 状态 | 无状态 | 有状态 |
| 节点职责 | 计算（函数） | 行为（动作） |
| 扩展方式 | 注册新函数节点 | 注册新行为节点 |
| 适用场景 | 伤害计算、条件判断、数值处理 | 技能释放、AI行为、关卡流程 |
| 与 Unity 类比 | `Func<T1, T2, TResult>` 委托 | `MonoBehaviour` 组件 |

> **两图通过明确的契约交互，构成一个完整的、图灵完备的游戏逻辑系统。逻辑图只负责"判断和计算"，控制图只负责"执行和调度"。逻辑图不能编辑控制图，控制图可以自由调用逻辑图。**
